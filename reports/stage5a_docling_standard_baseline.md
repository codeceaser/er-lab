## 6. Known Docling limitations discovered

These are genuine findings about Docling 2.114.0 / docling-core 2.87.1's
standard-local pipeline, discovered empirically while building and running
the adapter — not adapter defects, and not benchmark-correctness claims
(no manifest comparison was performed).

1. **DOCX exposes no page geometry at all.** `DoclingDocument.pages` is
   empty and every item's `.prov` is `[]` for DOCX input — page number,
   bounding box, and character span are all unavailable. Fallback: the
   adapter reads the source `.docx` file's own section `page_width`/
   `page_height` via `python-docx` (already a repository dependency, used
   here only for this one structural field) and uses it as
   `CanonicalUnit.width`/`height`, since the frozen canonical contract
   requires `width > 0`/`height > 0` and the adapter is not permitted to
   invent a fake Letter/A4 default. Every DOCX conversion records a
   `docling_page_geometry_unavailable` diagnostic when this happens. Every
   canonical element from a DOCX source therefore has `bbox=None`.

2. **PDF heading-level classification did not distinguish nesting depth
   for the parity fixture.** All three headings in `PARITY_001.pdf`
   (a title-level heading and two section headings) were assigned
   `level=1` uniformly by Docling's layout model — the manifest's
   intended title/H2 distinction was not recovered. By contrast, the same
   semantic headings in `PARITY_001.docx` were correctly assigned levels
   1/2/3 (Word's own paragraph-style metadata, read directly by Docling's
   DOCX backend, rather than inferred from visual layout).

3. **PPTX title/section-header shapes are not classified as headings at
   all.** `PARITY_001.pptx` produced **zero** `SECTION_HEADER`/`TITLE`
   labeled items — the same heading text present in the PDF/DOCX versions
   was returned as plain body text. All three parity-format headings
   became `CanonicalHeading`s for PDF and DOCX; none did for PPTX.

4. **Picture-to-caption linking works for PDF, not for DOCX/PPTX.**
   `PictureItem.captions` was correctly populated for `PARITY_001.pdf`
   (one `CanonicalCaption` correctly linked to the picture). For
   `PARITY_001.docx` and `PARITY_001.pptx`, `PictureItem.captions` was an
   empty list — the caption paragraph ("Figure 1: Escalation flow...")
   was mapped as an ordinary `CanonicalParagraph` instead, with no link to
   the picture at all.

5. **DOCX did not preserve multi-level nested-list parent/child
   structure.** `STRESS_DOCX_001.docx`'s intentionally 3-level nested list
   (5 items) was returned by Docling as **3 flat, sibling** `GroupItem`
   (`label=list`) containers, all sharing one parent heading, with
   `ListItem.level` always `None` and no per-item indent signal available
   anywhere in the public API. The mapper's ancestry-walk
   (`_walk_list_ancestry`) is real, general structural logic — it
   correctly computes `indent_level > 0` and `parent_block_id` when
   Docling's own output for a document genuinely nests list groups inside
   list items (proven directly in
   `tests/test_docling_standard_mapper.py::test_list_item_indentation_and_parent_relationship`
   using a hand-constructed nested `DoclingDocument`) — it is not a bug
   in the walk, it is Docling's real DOCX output for this fixture. Result:
   all 5 items landed at `indent_level=0`/`parent_block_id=None`, split
   across 3 distinct `list_id`s instead of the one the manifest's authors
   likely had in mind.

6. **OCR-origin evidence for individual text items is entirely
   structural, never explicit.** `TextItem.source` is `[]` for every text
   item observed in every fixture, regardless of whether that text came
   from native page content or from RapidOCR. The only usable signal is
   tree position: a `TextItem` nested directly under a `PictureItem` (and
   not that picture's caption) reliably indicates OCR-of-an-image-region
   in every fixture tested (`PARITY_001.pdf`'s 3 diagram OCR tokens,
   `STRESS_CHART_001.pdf`'s 9 chart-label OCR tokens) — but body-level
   OCR text (the whole-page OCR pass in `STRESS_SCANNED_001.pdf`, which
   has no picture wrapper) has no distinguishing signal at all and is
   mapped as an ordinary `CanonicalParagraph`, per the Stage 5A
   instruction to never fabricate `OcrAnnotation` provenance beyond what
   the parser evidences.

None of these are canonical-model or chunking-layer defects — the frozen
contracts (`ingestion_bench.canonical`, `ingestion_bench.chunking`) were
not modified to accommodate any of them, per the Stage 5A gate.
